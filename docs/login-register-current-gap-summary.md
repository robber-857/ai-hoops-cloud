# 登录与注册待完善功能总结

## 1. 文档目的

本文档基于当前仓库代码现状，梳理登录、注册、验证码发送、鉴权和会话管理还需要补齐的功能，作为后续开发清单。

适用范围：

- 前端：`web/src/components/auth/*`、`web/src/store/authStore.ts`、`web/src/services/auth.ts`
- 后端：`server/app/api/v1/auth.py`、`server/app/services/auth_service.py`
- 数据模型：`server/app/models/user.py`、`server/app/models/user_session.py`、`server/app/models/verification_code.py`

## 2. 当前已经具备的基础

目前项目已经有一套“认证骨架”，但还没有形成完整闭环：

- 已有登录页和注册页
- 已有后端认证路由
- 已有用户表、验证码表、会话表
- 已有密码哈希和 JWT 生成能力
- 已有手机号登录、邮箱登录、账号密码登录的接口定义
- 已有注册、登录、找回密码、刷新 token 的基础 schema

也就是说，当前更像是“页面和接口占位已搭好”，但生产可用的登录注册流程还没有真正完成。

## 3. 注册流程当前缺口

### 3.1 注册验证码发送未真正实现

当前 `POST /auth/register/send-code` 只是直接返回成功和过期时间，没有真正完成以下动作：

- 生成随机验证码
- 保存验证码记录到 `verification_codes`
- 记录验证码场景为注册
- 设置过期时间、发送时间、状态
- 通过短信或邮件服务真实发送验证码

补充说明：

- 现在接口只返回 `target: payload.phone_number`
- 即使请求里带了 `email`，也没有独立的邮箱验证码发送闭环

### 3.2 注册时未校验手机号验证码

当前注册接口 `POST /auth/register` 会直接创建用户，但没有校验：

- `phone_code` 是否存在
- `phone_code` 是否正确
- `phone_code` 是否过期
- `phone_code` 是否已使用
- `phone_code` 错误次数是否超限

因此现在的“手机号验证码”只是表单字段，不是真正的注册门槛。

### 3.3 邮箱验证未完成

当前注册类型里有 `email` 和可选的 `email_code`，但实际流程并未完成：

- 前端注册页没有邮箱验证码输入和发送入口
- 后端没有真正校验 `email_code`
- 用户注册后不会把 `is_email_verified` 设为已验证

如果产品要求“邮箱可选但填写后必须验证”，这一块还需要完整补齐。

### 3.4 用户验证状态没有在注册闭环中更新

`users` 表已经有：

- `is_phone_verified`
- `is_email_verified`

但当前注册成功后：

- 手机验证状态不会置为 `true`
- 邮箱验证状态不会置为 `true`

因此用户资料层已经预留了验证状态，但业务层没有真正接起来。

### 3.5 前端保留了开发测试验证码逻辑

当前注册页点击“发送验证码”后，会直接把本地验证码写成固定值：

- 前端自动回填 `phone_code = "1234"`
- 页面提示使用本地测试码

这类逻辑只能用于本地联调，正式环境必须移除。

### 3.6 注册输入校验还缺业务规则

目前主要只有基础长度校验，还缺少更贴近业务的校验：

- 手机号格式规范化
- 邮箱唯一性冲突时更明确的提示
- 用户名可用性规则
- 密码强度规则
- 是否必须同时绑定手机号和邮箱的产品规则

## 4. 登录流程当前缺口

### 4.1 账号密码登录中的验证码未真正校验

当前 `POST /auth/login/password-code` 会校验用户名和密码，但不会校验 `verify_code`。

也就是说：

- 接口要求传 `verify_code`
- 但服务端实际上忽略了它

这会造成前后端表面上有“密码 + 验证码登录”，实际上只有密码在生效。

### 4.2 账号密码登录缺少验证码发送入口

登录页的密码模式里有 `verify_code` 输入框，但当前没有对应的发送验证码按钮，也没有对应的发送接口被前端调用。

这意味着密码登录模式存在产品和实现不一致：

- 如果要做“密码 + 二次验证码”，需要补发送接口和校验逻辑
- 如果只想做“用户名 + 密码登录”，则应删除这个验证码字段

这个点需要先明确产品方案。

### 4.3 手机验证码登录未真正校验验证码

当前 `POST /auth/login/phone` 只会根据手机号查用户，不会校验：

- 验证码是否正确
- 验证码是否过期
- 验证码是否属于登录场景
- 验证码是否已使用

目前相当于“只要手机号存在就能登录”。

### 4.4 邮箱验证码登录未真正校验验证码

当前 `POST /auth/login/email` 只会根据邮箱查用户，不会校验：

- 邮箱验证码是否正确
- 是否过期
- 是否已使用
- 是否属于登录场景

目前相当于“只要邮箱存在就能登录”。

### 4.5 手机号/邮箱验证码发送未真正实现

当前：

- `POST /auth/login/phone/send-code`
- `POST /auth/login/email/send-code`

都只是返回成功消息，没有完成真实发送闭环：

- 未生成随机验证码
- 未写入验证码表
- 未校验发送频率
- 未接入短信或邮件服务

### 4.6 前端登录页保留了开发测试码自动回填

当前登录页里：

- 手机登录发送后会自动填入 `123456`
- 邮箱登录发送后会自动填入 `123456`

这同样属于本地联调用逻辑，正式环境必须删除。

## 5. 鉴权与会话管理当前缺口

### 5.1 登录成功后没有建立真实服务端会话

当前后端登录成功后只返回：

- `access_token`
- `refresh_token`
- `user`

但没有：

- 创建 `user_sessions` 记录
- 存储 refresh token 哈希
- 绑定设备、浏览器、IP 信息
- 设置 `HttpOnly` session cookie

虽然项目里已经有 `user_sessions` 模型，但现在完全没有用起来。

### 5.2 `/auth/me` 还没有实现

当前 `GET /auth/me` 直接返回 `501 Not Implemented`。

这会直接影响：

- 页面刷新后恢复登录态
- 前端判断当前用户是否已登录
- 受保护页面初始化
- 服务端可信登录态建立

### 5.3 `/auth/logout` 还没有真正退出

当前 `POST /auth/logout` 只是空返回，没有做：

- 服务端撤销会话
- 失效 refresh token
- 清理 cookie

这意味着“退出登录”目前没有真正的服务端语义。

### 5.4 refresh token 机制还不完整

当前 `POST /auth/refresh` 能解 JWT，但还没有做到：

- refresh token 入库追踪
- refresh token 撤销
- refresh token 轮换
- refresh token 与 `user_sessions` 绑定
- 失效后阻止继续刷新

现在更像是“可解析 token”，不是“完整会话系统”。

### 5.5 缺少统一鉴权依赖

后端当前只有数据库依赖 `get_db`，还没有真正接好：

- 从请求中提取当前登录态
- 校验 cookie 或 bearer token
- 获取当前用户
- 为受保护接口统一返回 `401`

因此现阶段认证接口之外的业务接口，还没有完整接入统一鉴权边界。

### 5.6 前端登录态仍然是本地伪状态

当前前端 `authStore` 主要做了两件事：

- 把 token 和用户信息持久化到 `localStorage`
- 写入一个前端可写的 `ai-hoops-authenticated=1` cookie

问题是：

- 这个 cookie 不是 `HttpOnly`
- 这个 cookie 不能代表真实服务端登录态
- 页面是否“已登录”仍然偏前端自判

这只能作为临时 UI 状态，不能作为正式鉴权依据。

### 5.7 前端请求层还没有接入 cookie 会话方案

当前 `web/src/services/client.ts` 里 `fetch` 没有统一携带：

- `credentials: "include"`

因此即使后端后续切到 cookie 会话，前端请求层也还需要同步改造。

### 5.8 中间件还没有真正做页面保护

当前 `web/middleware.ts` 直接 `NextResponse.next()`，没有对以下页面做任何登录拦截：

- `/pose`
- `/pose-2d/*`
- 其他需要登录后访问的功能页

也就是说，页面级登录保护目前基本未接入。

## 6. 找回密码相关缺口

虽然这次重点是登录和注册，但当前找回密码也有同类问题：

- `POST /auth/password/send-reset-code` 只是占位返回
- `POST /auth/password/reset` 不校验验证码，只要能找到用户就能改密码

这部分和登录注册共享同一套验证码能力，建议一起整改。

## 7. 验证码系统当前缺口

虽然数据库里已经有 `verification_codes` 表，但业务层还没有真正落地这些能力：

- 生成随机验证码
- 区分验证码场景：注册、登录、找回密码
- 区分手机号和邮箱目标
- 验证码加密或哈希存储
- 过期校验
- 已使用校验
- 错误次数累计
- 最大尝试次数限制
- 重发频率限制
- 同一目标短时间限流
- 验证成功后置为 `used`
- 清理过期验证码

换句话说，数据结构已经有了，但验证码服务还没实现。

## 8. 环境配置当前缺口

目前环境变量中已经有数据库、JWT、CORS 配置，但没有看到登录注册闭环真正需要的配置项，例如：

- 短信服务商配置
- 邮件服务商配置
- 验证码有效期配置
- 验证码发送频率限制配置
- 登录失败锁定阈值配置
- 会话 cookie 配置
- session 过期时间配置

如果要把登录注册做成可上线能力，这些配置也需要补齐。

## 9. 建议优先级

建议按下面顺序推进：

### P0：先补安全闭环

- 实现验证码真实发送与存储
- 注册校验手机号验证码
- 登录校验手机/邮箱/密码登录对应验证码
- 实现 `/auth/me`
- 实现真实 logout
- 建立服务端 session 或至少可信会话机制

### P1：再补前端接入

- 删除前端测试验证码自动回填
- 登录态改为以服务端结果为准
- 请求层统一支持 `credentials: "include"`
- 页面和接口接入真实鉴权判断

### P2：最后补生产能力

- 验证码限流
- 登录失败风控
- 会话管理
- 设备管理
- 审计日志
- 更完整的错误提示和用户体验

## 10. 一句话结论

当前登录和注册已经完成了页面、接口、表结构的“骨架搭建”，但验证码发送、验证码校验、真实鉴权、会话管理、退出登录、当前用户识别都还没有闭环。

如果要进入可上线状态，最优先补的是：

- 真实验证码系统
- 真实登录校验
- 真实服务端会话
- 真实页面与接口鉴权
