# AI Hoops Cloud 生产安全迁移方案

## 1. 背景

当前项目已经具备登录、注册、验证码、上传、分析等核心流程，但整体认证与会话体系仍偏开发态，存在以下问题：

- 前端曾依赖 `localStorage` 或前端可写 cookie 作为登录态依据。
- 受保护页面可以通过前端跳转守卫处理，但真正的安全边界应该在服务端和受保护接口。
- 登录、注册、验证码流程仍存在从“功能可用”走向“生产安全”的迁移工作。
- 上传与分析链路如果只靠页面拦截，而不是服务端鉴权，仍然可能被绕过。

本方案的目标是把项目升级为适合正式环境的“服务端可信认证体系”。

## 2. 迁移目标

### 2.1 认证目标

- 前端不再持有“最终可信认证真相”。
- 登录状态以服务端签发并验证的会话为准。
- 页面访问、接口调用、上传签名都必须经过服务端鉴权。
- 登录 cookie 不暴露给前端 JavaScript。

### 2.2 安全目标

- 防止用户伪造前端 cookie 直接进入受保护页面。
- 防止仅靠修改前端状态绕过鉴权。
- 减少 XSS、CSRF、暴力破解、验证码滥用、会话劫持风险。
- 为后续审计、风控、会话撤销和多设备管理留出结构空间。

## 3. 推荐架构

## 3.1 首选方案：服务端 Session + HttpOnly Cookie

对于当前这个浏览器访问的网站，推荐优先使用“服务端 Session + HttpOnly Cookie”。

原因：

- 比纯 JWT 浏览器方案更适合传统 Web 应用。
- 更容易做会话撤销、踢下线、设备管理。
- 前端不需要读取 token，减少暴露面。
- 中间件、服务端渲染、接口鉴权都更自然。

建议模型：

- 登录成功后，后端创建 `user_sessions` 记录。
- 后端签发一个只包含 session 标识的 cookie。
- cookie 设置为：
  - `HttpOnly`
  - `Secure`
  - `SameSite=Lax` 或更严格的 `Strict`
  - `Path=/`
  - 合理 `Max-Age` 或 `Expires`
- 浏览器后续请求自动携带 cookie。
- 后端每次请求根据 cookie 查 session 是否存在、是否有效、是否撤销、是否过期。

## 3.2 JWT 是否需要

JWT 不是当前项目上线的前置条件。

只有在以下场景中才建议优先考虑 JWT：

- 移动端 App 与 Web 共用认证体系
- 多个独立服务之间做无状态鉴权
- 第三方 API 接入
- 明确需要跨服务分发 access token

如果后续确实要用 JWT，建议采用：

- 短生命周期 access token
- 可撤销、可轮换的 refresh token
- 服务端严格验签、验过期、验 `iss` / `aud` / `type`
- 浏览器端仍尽量通过 `HttpOnly` cookie 承载，而不是放在 `localStorage`

## 4. 当前项目对应的高风险点

### 4.1 前端风险

- `web/src/store/authStore.ts`
  - 目前前端 store 可持有登录态信息。
  - 这类状态适合做 UI 展示，不适合作为最终安全依据。
- `web/middleware.ts`
  - 中间件只能基于可信 cookie 做拦截。
  - 如果 cookie 本身是前端可伪造的，守卫只是“体验层”，不是“安全层”。
- `web/src/components/auth/LoginClient.tsx`
  - 登录成功后的行为应与服务端真实 session 建立绑定。

### 4.2 后端风险

- `server/app/api/v1/auth.py`
  - 当前返回的是 token 型响应，但 `/me` 仍是占位实现。
- `server/app/services/auth_service.py`
  - 当前逻辑仍偏 JWT 登录流程，没有完成服务端 session 作为主认证依据的收口。
- `server/app/api/deps.py`
  - 目前只有数据库依赖，没有“当前登录用户”依赖。
- 上传、报告、分析相关 API
  - 如果未统一接入鉴权依赖，就存在未授权访问风险。

### 4.3 验证码与风控风险

- 注册、登录、找回密码验证码接口需要限流。
- 正式环境不得保留测试验证码、自动回填验证码或开发快捷登录逻辑。

## 5. 推荐的生产安全控制项

## 5.1 认证与会话

- 使用服务端 session 作为最终认证依据
- 登录成功时创建 session 记录
- cookie 使用 `HttpOnly`、`Secure`、`SameSite`
- 支持 session 撤销、过期、单设备或多设备管理
- logout 必须服务端删除 session，而不是只清前端状态

## 5.2 接口安全

以下接口必须做服务端鉴权：

- 当前用户信息接口
- 视频上传接口
- 上传签名接口
- 分析报告读取接口
- 报告导出接口
- 任何写数据库的接口

## 5.3 攻击防护

- 登录、注册、验证码、找回密码接口限流
- 登录失败次数控制
- 验证码发送频率限制
- CSRF 防护
- XSS 防护
- 输入校验与输出编码
- 上传文件类型、大小、扩展名、内容校验
- 敏感日志脱敏

## 5.4 基础设施

- 生产环境全站 HTTPS
- Cookie 强制 `Secure`
- 密码采用 `Argon2` 或 `bcrypt`
- 环境变量与密钥不入库、不进前端
- 增加审计日志与异常登录监控

## 6. 推荐迁移原则

- 优先保证“服务端可信”
- 前端状态只作为展示层状态
- 页面守卫和接口守卫同时推进
- 上传能力一定要在服务端鉴权后放开
- 先收敛认证，再做风控增强

## 7. 分阶段实施建议

### 阶段一：认证收口

- 后端登录成功后下发 `HttpOnly` session cookie
- 完成 `/auth/me`
- 完成统一“当前用户”依赖
- 中间件只认服务端可信 cookie

### 阶段二：接口保护

- 上传、报告、分析接口接入鉴权依赖
- 未登录访问统一返回 `401`
- 前端根据 `401` 跳回登录

### 阶段三：风控与安全加固

- 验证码、登录、注册限流
- CSRF 防护
- 安全日志
- 会话撤销与设备管理

## 8. 上线前检查清单

- 删除所有测试验证码逻辑
- 删除前端自动回填验证码逻辑
- 删除调试登录 bypass
- 登录 cookie 改为 `HttpOnly`
- 受保护页面与接口都已接入鉴权
- 上传链路必须服务端验权
- 所有生产环境走 HTTPS
- 关键接口已限流
- 密码哈希算法符合生产标准
- 审计与异常监控已启用

## 9. 结论

如果你后续要真正防止用户伪造登录、绕过保护、滥用上传能力，核心不是“前端路由守卫做得更复杂”，而是把认证彻底收回服务端。

对当前项目来说，最合适的生产方向是：

1. 以服务端 session 为主
2. 以 `HttpOnly` cookie 为浏览器承载方式
3. 以服务端接口鉴权为真正安全边界
4. 在此基础上补限流、CSRF、XSS、审计和上传安全
